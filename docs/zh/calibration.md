# 校准报告 —— "89.7%" 到底是怎么测出来的

RootPilot 引用的每一个准确率数字,都来自同一套固定基准:**29 个故障场景 + 2 个健康对照**,在真实主机上真实注入、走产品标准诊断流程打分。本页公开完整场景清单、评分标准和逐场景得分——包括我们**没有**拿到满分的场景。

> **38 条只读命令 · 29 个标准故障场景 · 根因准确率 89.7%(26.0/29)· 4 次健康对照零误报。**

## 方法

- **真实注入,不是 mock。** 每个场景都在真实的 Ubuntu 22.04 主机上注入——真实容器、loop 设备、连接洪泛、停掉的系统服务——先验证产出预期证据,再诊断,最后清理。没有预制日志,没有合成数据。
- **评分标准先于测试写定。** 每个场景都预先写好预期根因和 1 / 0.5 / 0 评分要点。所有跑测全部计分,不挑样本、不重摇。
- **走产品原路径,不开小灶。** 诊断走标准流程——基础采集 → 场景组采集 → 至多一轮追问——用的是每个用户拿到的同一套[38 条只读命令白名单](security.md#5-完整的-38-条命令白名单),没有任何场景专属提示。
- **经济型默认模型。** 下列得分全部由 DeepSeek(`deepseek-v4-flash`)产出——最便宜的合理默认。它失分的地方,更强的模型在相同证据上得分更高(见[失分在哪里](#失分在哪里));BYOK 意味着这个取舍由你决定。
- **健康对照是硬门槛。** 在健康主机上虚构问题,整轮校准直接判负。两个健康场景在两轮校准中各跑一遍:**4/4 干净,零误报。**

## 评分标准

| 得分 | 含义 |
|---|---|
| **1** | 根因命中该场景预写标准——机理正确,且引用了正确的证据 |
| **0.5** | 方向对,但机理错误或不完整 |
| **0** | 漏判或误判 |

## 失分在哪里

我们公开失分项,因为它们比得分项更能说明问题:

- **`egress-blocked`(0.5)** —— 证据*已经*采到(探针 exit 7),默认模型却把它合理化为"目标侧限制,非本机故障"。同证据对照实验中,Claude 系模型得 1 分。
- **`service-stopped`(0)** —— `cron: inactive` 明明白白在证据里,默认模型仍判主机健康。
- **`deleted-file-disk`(0.5)** —— 找到了满盘,漏掉"已删除仍占用"的机理;始终没有主动调用能揭穿它的追问命令。
- **`docker-cache-bloat`(0)** —— 尚余 28 GB 空闲时,模型(情有可原地)判无故障,也没查 Docker 自身的磁盘占用。弱信号的卫生型场景。

四个失分项的共同模式:*采集层*把证据交上来了,经济型默认模型没能追查或采信**沉默证据**——静默失败的探针、缺席而非报错的服务。这是模型能力的取舍,不是采集缺口。这也正是 RootPilot 做成 BYOK 的原因:换更强的模型,这类场景直接改善;或保留便宜默认,吃下其余 25 个它已经答对的场景。

## 诚实备注

- **基准变难后,主数字是降的。** 最初的 24 场景集在采集与 prompt 迭代后达到 93.75%;加入 5 个更难的场景后,总盘降为 89.7%(26.0/29)。我们引用低的那个。
- **两轮校准之间改了什么:** 一处采集探针 bug 修复(egress 探针报错了退出码)、两条白名单新增(容器内监听列表、系统服务状态——均已列入 [security.md](security.md))、一条 prompt 规则(沉默证据默认按需上报处理,不得合理化跳过)。场景定义与评分标准自始至终未放松。
- **`clock-drift` 为软注入**:漂移通过应用日志与 token 校验失败呈现,而非真改主机时钟——在共享测试机上真拨时钟会连带打坏 TLS 和 NTP。被评分的证据路径不变。
- **模型与轮数:** 所有诊断至多一轮追问采集——与产品出厂限制一致。

## 复现它的精神

采集层——同一套只读白名单思路——已开源为 MCP server:[rootpilot-mcp](https://github.com/Easton-OU/rootpilot-mcp)。把它指向一台可随意折腾的虚拟机,注入上表中任何一个故障,在你自己的 LLM 客户端里看看原始证据长什么样。
